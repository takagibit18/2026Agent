## 三、关键技术选型判断原则

> 三方一致认定"技术选型判断力"为元能力，以下为共同认可的决策框架。

### 3.1 RAG vs Fine-tuning vs Prompt Engineering 决策树

```
问题：如何让 LLM 具备特定领域知识或行为？
│
├─ 知识是否频繁更新？
│   ├─ 是 → 优先 RAG（知识库可实时更新，无需重新训练）
│   └─ 否 → 继续判断 ↓
│
├─ 是否需要特定输出格式/风格/语气？
│   ├─ 是，且样本充足（>500条） → 考虑 Fine-tuning
│   ├─ 是，但样本少 → 先用 Few-shot Prompt Engineering
│   └─ 否 → 继续判断 ↓
│
├─ 基础模型能力是否足够？
│   ├─ 是 → Prompt Engineering（成本最低，迭代最快）
│   └─ 否（如专业术语、特定推理模式）→ Fine-tuning
│
└─ 综合考量
    ├─ 预算有限 → Prompt Engineering > RAG > Fine-tuning
    ├─ 延迟敏感 → Fine-tuning（无检索延迟）> Prompt Engineering > RAG
    └─ 知识量大（>100万字） → RAG 必选
```

### 3.2 向量数据库选型原则


| 场景                  | 推荐                            | 理由                                   |
| ------------------- | ----------------------------- | ------------------------------------ |
| 本地开发/原型验证           | **Chroma**                    | 零配置，纯 Python，适合快速迭代                  |
| 生产环境，中小规模（<1000万向量） | **Qdrant**                    | Rust 实现性能好，支持 payload 过滤，Docker 部署简单 |
| 大规模（亿级向量）           | **Milvus**                    | 分布式，生态成熟，国内大厂广泛使用                    |
| 已有 Elastic 基础设施     | **Elasticsearch + kNN**       | 复用现有投资，减少运维成本                        |
| 需要全托管，快速上线          | **Pinecone / Weaviate Cloud** | 减少运维负担，适合 MVP 阶段                     |


### 3.3 LLM 模型选型原则


| 场景          | 推荐策略                                    |
| ----------- | --------------------------------------- |
| 复杂推理、长上下文理解 | GPT-4o / Claude 3.5 Sonnet              |
| 高并发、成本敏感    | GPT-4o-mini / Claude Haiku / 通义千问-Turbo |
| 数据不出境（合规要求） | 本地部署开源模型（Qwen2.5 / DeepSeek-V3）         |
| 代码生成        | Claude 3.5 Sonnet / DeepSeek-Coder-V2   |
| 中文场景为主      | 通义千问 / 文心 4.0 / 智谱 GLM-4                |
| 推理任务（数学/逻辑） | DeepSeek-R1 / o1-mini                   |


### 3.4 Agent 架构选型原则


| 场景            | 推荐架构                 | 理由                |
| ------------- | -------------------- | ----------------- |
| 单轮问答          | 直接 LLM API           | 无需 Agent，避免过度工程化  |
| 多步骤、工具调用（<5步） | **ReAct Agent**      | 简单、可控、易调试         |
| 复杂任务分解（>5步）   | **Plan-and-Execute** | 先规划再执行，避免短视决策     |
| 多角色协作         | **Multi-Agent 框架**   | 仅在单 Agent 能力上限时引入 |
| 长耗时后台任务       | **异步 Agent + 消息队列**  | 解耦请求和执行，提升用户体验    |

## 五、面试必备问题清单

> 能流畅回答以下 10 个问题，大厂 Agent 岗位 Offer 概率大幅提升。

### RAG 相关（必考）

1. **你的 RAG 系统检索准确率是多少？你怎么评估和提升的？**
  - 期望答案：有具体指标（RAGAS Faithfulness/Relevancy），有优化前后对比，能说出优化方法（混合检索/Reranker/分块策略调整）
2. **纯向量检索有什么局限性？你是如何解决的？**
  - 期望答案：向量检索对精确关键词（人名/型号）效果差，混合检索（BM25+向量+RRF）是解决方案
3. **Reranker 的原理是什么？为什么要用两阶段检索？**
  - 期望答案：Bi-Encoder 做粗排（速度快），Cross-Encoder 做精排（精度高），两阶段是成本/效果的最优解

### Agent 相关（必考）

1. **ReAct 和 Plan-and-Execute 的区别是什么？你什么时候选哪个？**
  - 期望答案：ReAct 适合短任务（实时反馈），Plan-and-Execute 适合长任务（避免短视），能举具体例子
2. **你的 Agent 遇到工具调用失败怎么处理？**
  - 期望答案：重试策略（指数退避）、降级方案、错误信息反馈给 LLM 让其重新规划

### 工程化相关（必考）

1. **你的 Agent 服务最高支持多少并发？瓶颈在哪里？**
  - 期望答案：有压测数据，能分析瓶颈（LLM API 延迟/向量检索/网络 I/O），有优化方案
2. **如果 LLM API 突然不可用，你的系统如何降级？**
  - 期望答案：备用模型切换、缓存结果返回、友好错误提示、告警通知

### 原理相关（大厂必考）

1. **KV Cache 是什么？它如何影响 LLM 推理性能？**
  - 期望答案：缓存 Attention 计算的 Key/Value 矩阵，避免重复计算，显著降低自回归生成延迟
2. **为什么长文本会出现"Lost in the Middle"问题？**
  - 期望答案：Attention 机制对序列中间位置的关注度低于开头和结尾，是模型训练分布的问题

### 系统设计（大厂必考）

1. **设计一个支持 10 万用户的企业知识库问答系统，你会怎么做？**
  - 期望答案：负载均衡 + 无状态 API 集群 + Redis 会话存储 + Milvus 向量库集群 + 消息队列异步处理 + 监控告警，能讨论成本和延迟权衡
