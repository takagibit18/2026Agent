# 中国本科大三学生冲大厂 Agent 开发 / LLM 应用工程师

## 结论

本方案由 3 个 subagent 交叉讨论、互相评审并收敛后形成，最终一致同意：

- 主攻岗位应定位为 `LLM 应用工程师 / Agent 开发工程师 / AI 应用后端工程师`
- 不建议把 3 个月主线放在 `大模型算法研究 / 重训练 / 重微调 / 复杂多智能体`
- 求职最优路径是：`Python 工程能力 + 后端服务开发 + RAG + Tool Calling / Workflow + 评测 + 部署 + 面试表达`

一句话总结：

> 对中国本科大三学生来说，最有胜率的路线不是“证明自己会造模型”，而是“证明自己能把 LLM 做成一个可信、可部署、可评测的小型工程系统”。

---

## 一、技术栈优先级排行

以下排序按“3 个月内对大厂实习/校招面试通过率的边际收益”排序，而不是按技术先进程度排序。

### 第 1 名：`Python` 工程能力

必须掌握：

- 项目结构化组织
- 常见数据处理
- 异步基础
- 日志、配置、异常处理
- 能写接口、脚本、评测工具

原因：

- 几乎所有 LLM 应用工程岗位都默认以 `Python` 为核心语言
- 这是 RAG、Agent、后端服务、评测脚本的共同底座

### 第 2 名：后端开发基础 `FastAPI + HTTP + API 设计`

必须掌握：

- `FastAPI`
- 路由、请求响应、状态码、鉴权基础
- 超时、重试、限流、错误处理
- 基本服务拆分思维

原因：

- 面试官会默认把你视作“AI 应用工程岗”，而不是“Prompt 使用者”
- 不会服务化，项目就容易停留在 demo 层

### 第 3 名：数据与存储 `SQL + Redis`

必须掌握：

- 基础表设计
- 查询与索引常识
- 会话状态
- 缓存
- 任务状态管理

原因：

- 真正的 LLM 项目绕不开结构化数据、缓存和状态管理
- 面试里经常会问为什么这里用数据库、那里用缓存

### 第 4 名：`RAG` 全链路

必须掌握：

- 文档解析
- chunking
- embedding
- 向量检索
- rerank 基础
- 生成与拒答
- 基础评测

原因：

- 这是当前中国大厂最常见、最有业务落地价值的 LLM 应用能力
- 也是本科生最容易做出“简历级项目”的主线方向

### 第 5 名：`Prompt + Structured Output + Tool Calling`

必须掌握：

- system prompt 设计
- few-shot 基础
- 结构化输出约束
- function/tool schema
- 工具失败时的回退逻辑

原因：

- 企业场景更看重“稳定输出”和“可控调用”，不是玄学式提示词
- Tool Calling 是 Agent 落地的基础能力

### 第 6 名：`Agent Workflow / 单 Agent 状态编排`

必须掌握：

- 什么时候该用 workflow
- 什么时候不该上 agent
- 状态流转
- 步数限制
- 失败重试与回退

原因：

- 大厂岗位名虽然常写 Agent，但实质多数是“受控工作流 + 工具编排”
- 单 Agent 做稳，比复杂多 Agent 更有求职价值

### 第 7 名：评测与实验设计

必须掌握：

- 构建小型测试集
- 基线对比
- 任务成功率
- 检索命中情况
- 人工抽样评审
- 错误案例分析

原因：

- “我觉得效果不错”在面试中没有说服力
- 能证明优化有效，是 demo 和工程项目的分水岭

### 第 8 名：成本控制与可观测性

必须掌握：

- token 成本统计
- 延迟记录
- 调用日志
- prompt/version 留痕
- 失败归因
- 缓存与模型路由基础

原因：

- 招聘经理越来越看重“能不能落地”和“值不值得上线”
- 成本、稳定性、可观测性是工程成熟度的重要标志

### 第 9 名：工程部署 `Linux + Git + Docker`

必须掌握：

- `Git` 基本协作
- `Linux` 常用命令
- `Docker` 打包
- 基础上线与环境复现

原因：

- 没有部署能力，项目可信度会打折
- 至少要能让面试官相信你的系统不只是在本地 notebook 中运行

### 第 10 名：前端最小展示能力

建议掌握：

- 简单页面或聊天界面
- Demo 展示
- 基础交互

原因：

- 不要求深挖前端
- 但有一个可展示页面，会显著提升项目完成度与作品集观感

## 二、明确延后或降优先级的方向

以下内容不是没价值，而是 **不适合当前 3 个月作为主线**：

- `LoRA / SFT / 全量微调`
- `复杂多智能体`
- `vLLM / 推理内核深度优化`
- `K8s`
- `MCP` 深度开发
- `多模态` 作为主线项目
- `论文复现` 作为核心成果
- `从零训练模型`

原因：

- 对本科大三、目标大厂应用工程岗位的人来说，投入大、见效慢、面试转化率不高

## 三、三个月学习规划

总原则：

- 以 `1 个重项目 + 1 个轻项目/增强模块` 为主，不要贪多
- 算法题、八股、项目表达从第 1 周开始并行，不要最后补
- 所有学习内容都要回到“是否提升简历可信度与面试通过率”来判断

### 第 1 阶段：第 1-3 周，打底与最小闭环

目标：

- 从“会调 API”进化到“能做基础 AI 应用服务”

#### 第 1 周

学习内容：

- `Python` 工程化
- LLM API 调用
- Prompt 基础
- Structured Output

交付物：

- 一个最小 LLM 小工具
- 10 个 prompt/structured output 实验记录

#### 第 2 周

学习内容：

- `FastAPI`
- HTTP 接口设计
- `SQL` 基础
- 日志、配置、异常处理

交付物：

- 一个基础后端服务
- 简单 API 文档

#### 第 3 周

学习内容：

- `RAG` 基础链路
- 文档切分
- embedding
- 检索
- 基础评测

交付物：

- 一个基础知识库问答 demo
- 30 条左右测试样本

并行任务：

- 每周固定刷算法题
- 开始整理 Python/网络/数据库高频八股

### 第 2 阶段：第 4-6 周，主项目成型

目标：

- 做出一个真正能写进简历的主项目

建议选题二选一：

- 企业知识库助手
- 面向招聘/简历场景的求职助手

#### 第 4 周

学习内容：

- 项目立项
- 数据来源设计
- 文档处理流程
- 最小可演示链路

交付物：

- 主项目 v1
- README 初稿

#### 第 5 周

学习内容：

- 检索优化
- rerank
- 结构化输出稳定性
- 拒答与失败分析

交付物：

- 主项目 v2
- 一份评测表
- 一份失败案例分析

#### 第 6 周

学习内容：

- Tool Calling
- 单 Agent Workflow
- 状态流转
- 回退机制

交付物：

- 主项目加入工作流能力
- 架构图初版

并行任务：

- 继续算法题
- 每周做一次 3 分钟项目讲解练习

### 第 3 阶段：第 7-9 周，工程强化

目标：

- 把“能跑”升级成“可信、可讲、可部署”

#### 第 7 周

学习内容：

- 会话状态
- 缓存
- 重试
- 超时
- 限流
- 权限边界基础

交付物：

- 主项目工程增强版

#### 第 8 周

学习内容：

- 评测脚本完善
- token 成本统计
- 延迟分析
- 日志和 trace

交付物：

- 一套可复用评测脚本
- 优化前后对比结果

#### 第 9 周

学习内容：

- `Docker`
- 基础部署
- 演示页面
- Demo 视频

交付物：

- 至少 1 个可部署版本
- 完整 README
- 架构图与演示材料

并行任务：

- 继续算法题
- 整理项目追问清单

### 第 4 阶段：第 10-12 周，求职冲刺

目标：

- 把项目转化为简历、面试、投递成果

#### 第 10 周

学习内容：

- 简历定向改写
- 项目 STAR 表达
- 项目亮点提炼

交付物：

- 正式简历 v1
- 项目讲稿 v1

#### 第 11 周

学习内容：

- 模拟面试
- Python/网络/数据库/Redis 高频题
- 算法补位

交付物：

- 面试题清单
- 常见追问答法

#### 第 12 周

学习内容：

- 定向投递
- 按 JD 改写项目描述
- 作品集收尾

交付物：

- 最终简历
- 仓库主页/作品集
- 投递清单

## 四、推荐项目组合

### 主项目 1：RAG/知识库项目

推荐方向：

- 企业知识库问答
- 校园资料问答
- 开源项目文档助手

要体现的能力：

- 数据处理
- 检索与 rerank
- 拒答
- 评测
- 部署

### 主项目 2：Agent/Workflow 项目

推荐方向：

- 求职助手
- 资料分析助手
- SQL 数据分析助手

要体现的能力：

- Tool Calling
- 状态机式编排
- 失败回退
- 成本统计
- 任务链路可解释

### 轻项目/增强模块

推荐方向：

- 评测工具
- 日志观测面板
- Prompt/版本对比脚本

作用：

- 拉开和“只有 demo”候选人的差距

---

## 五、面试官最看重的 5 个点

1. 你是否有真实、可追问、可复盘的项目  
2. 你是否理解 RAG/Agent 的工程本质，而不只是会调框架  
3. 你是否能证明优化有效，而不是主观说“效果不错”  
4. 你是否具备基础后端能力，能把 LLM 做成服务  
5. 你是否能清楚表达 trade-off、失败案例和成本意识  

---

## 六、执行建议

### 每周时间分配建议

- `50%` 主项目建设
- `20%` 基础技术栈补强
- `15%` 算法题与基础八股
- `15%` 简历、表达、作品集、复盘

### 学习时的取舍标准

优先做：

- 能写进简历
- 能被面试验证
- 能沉淀成公开作品
- 能量化结果

尽量少做：

- 只看不做
- 只堆概念
- 只做聊天 demo
- 追热点但没有闭环

---

## 七、最终统一方案

3 个 subagent 最终一致同意的方案如下：

- 岗位定位：`LLM 应用工程 / Agent 开发 / AI 应用后端`
- 主线能力：`Python -> 后端 -> SQL/Redis -> RAG -> Prompt/Tool Calling -> Workflow -> Eval -> Cost/Observability -> Deploy`
- 项目策略：`1 个重项目 + 1 个轻项目/增强模块`
- 时间节奏：`12 周四阶段推进，项目、刷题、面试表达全程并行`
- 明确不进入主线：`重微调、复杂多智能体、K8s、MCP、推理内核、论文导向`

如果只能抓最关键的一句话：

> 先把自己训练成“会做 AI 应用系统的工程型候选人”，再考虑向模型训练或更前沿方向扩展。
